<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Express HTTP/WebSocket | All untrusted client traffic crosses here; CSP and Helmet headers harden it |
| Express → SQLite | DB write path runs through prepared statements only (no string concat) |
| New WebSocket client → io.on('connection') handler | Unauthenticated read-only channel; handler must only EMIT, never trust client-supplied data for state |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-03 | Tampering | Helmet CSP for WebSocket | mitigate | app.js CSP override adds `ws:` and `wss:` to `connect-src` so Socket.io upgrades are not silently blocked (Pitfall 1 from research). Verified via response-header inspection in Task 3 verify. |
| T-02-04 | Tampering | Emit before DB write | mitigate | This plan establishes the io infrastructure but does NOT emit from any write path yet — that is plan 02-03 Task 1 which inserts emits AFTER the existing DB INSERT/DELETE. DB-first invariant from Phase 1 is preserved by deferring all emit-on-write logic to 02-03. |
| T-02-Sx-1 | Spoofing | io.on('connection') unauthenticated socket | accept | TV WebSocket clients are read-only; server only emits computed state from the DB. Client-supplied socket data is never trusted (handler reads game id from DB query, not from socket). No client→server events accepted in this plan. |
| T-02-Inj | Injection | Migration ALTER TABLE | mitigate | Migration SQL is static string literals in db/index.js, never interpolated with user input. No injection surface. |
| T-02-Idem | Reliability | Repeated server starts | mitigate | try/catch on `'duplicate column name'` per D-12 — re-running the migration is a no-op. Verified by DB06. |
</threat_model>
